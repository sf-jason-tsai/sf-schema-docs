為什麼架構長這樣？
Thermos demo 需要「Steven 一鍵展示所有 AI 能力」，所以走 ConnectApi.EinsteinLLM 直接 wrap 而不是 Prompt Builder / Agentforce Bot：
1) demo org 保證跑得動（不用等 licence 佈署）
2) prompt 可以在 Apex 裡即時 tune，同一份程式 org 之間搬移
3) 走 Einstein Trust Layer（applicationName = PromptBuilderPreview），Zero Data Retention 一樣受保護
4) 功能上等於「自建的 Agentforce for Service」——LLM ＋ RAG（KM/Order/Account）＋ 8 個真動作。

Case Record Page · 8 個 AI 功能（真的呼叫 Einstein）

以下功能全在 ThermosCase360Controller.cls（2,374 行）。每個功能都有：（1）真 SOQL 拉 case + account + order 資料當 context，（2）組 prompt 呼叫 Einstein，（3）失敗時 fallback，（4）成功時 把結果寫回 Case 自訂欄位 並 留 CaseComment audit trail，（5）計入 THERMOS_Agent_Credits_Used__c。

01

Case AI 摘要（3 段結構化：問題／狀態／情緒）

ThermosCase360Controller.analyzeCase() · 呼叫 EinsteinService.generate

把 Case 主旨、描述、通路、狀態、優先度、客戶等級、近 3 則 CaseComment 全打包成 context，要求 Einstein 用嚴格三段格式輸出：

【問題】客戶要解決什麼（40 字） 【狀態】目前處理進度 ＋ 已用資源（40 字） 【情緒】客戶情緒判斷（平靜／困擾／生氣／滿意）＋ 依據（40 字）

結果寫回自訂欄位、在 UI 分三格顯示。以下是 CBUSDO 上抓出來的一筆真資料：

【問題】吳 淑芬於 電話 反映零件詢問相關問題。 【狀態】案件為「Closed」，AI 已加權評分並比對 KM。 【情緒】生氣 — 依主旨與描述關鍵字判斷。
寫回欄位用途
THERMOS_AI_Summary__c（LongTextArea）Case 頁 AI Summary Card 三欄顯示
THERMOS_AI_Summary_Short__c（Text 255）Case List View hover / Related List 用
THERMOS_Sentiment__c（Picklist）Case 頁 Hero + Priority Score 因子輸入
02

Case 情緒偵測

同上 analyzeCase() · sentiment 從摘要抽出 / fallback keyword rule

Einstein 產出摘要後，同一份文字裡抽出「生氣／困擾／平靜／滿意」四值，寫入 THERMOS_Sentiment__c。UI 上以 SLDS utility:priority / frown / smiley_and_people / emoji icon 對應。

為什麼不用 Einstein Case Classification (OOTB)：OOTB 的 Case Classification 需要歷史資料訓練，SDO 上沒足夠標記樣本；LLM 走 zero-shot prompt 效果更好，且產出「情緒 + 依據」比 OOTB 只給類別更有 demo 說服力。
03

Case 加權優先度評分（Priority Score）

scoreCasePriority() · 5 因子加權 + Einstein 一句話建議

先跑 規則式加權（會員等級 30% + LTV 20% + 情緒 25% + 緊急關鍵字 15% + 案件類型 10%），得 0-100 分，衍生 High / Medium / Low。

再把「總分 + 因子明細」餵給 Einstein，讓它生 40 字內的一句話建議：

你是 THERMOS 客服 Priority Advisor。以下為案件加權評分結果，請用繁體中文 40 字內給客服代理人一句處理建議： 案件：JNL-352 保溫瓶墊圈相容詢問（零件詢問） 客戶：白金會員 · LTV NT$ 82,400 情緒：困擾 總分：61/100 · 建議 Priority：High

然後 真的 UPDATE Case.Priority + Case.THERMOS_AI_Rating__c（1-5 星）、記 credit、在 CaseComment 留 audit：

[Agentforce Priority Score] 總分 61/100 · Medium → High。因子：會員 22 / LTV 12 / 情緒 12 / 緊急 0 / 類型 4。建議：30 分鐘內回覆 · 主動致電白金會員。
04

Case 自動分類 + 路由建議

classifyCase() · 7 類 + 8 隊列/專員路由表

用結構化 prompt 強制 LLM 用 4 行輸出：

CATEGORY: <零件詢問｜商品諮詢｜退換貨｜保固維修｜使用教學｜會員/點數｜訂單問題> CONFIDENCE: <0-100> REASON: <15 字內判斷理由> ALT: <其他可能類別，最多 2 個>

解析後：（1）寫回 THERMOS_Case_Category__c；（2）依 category 查路由表建議 Queue + Owner；（3）留 CaseComment audit：

[Agentforce Classification] 自動分類為「零件詢問」（信心 92%）· 建議路由至：零件專家隊列 / Steven Wang（MIS）。判斷依據：主旨提到墊圈料號。

Fallback：LLM 失敗時走 ruleClassify 關鍵字比對（信心固定 78%）。

05

Case AI 回覆草稿（RAG：KM + 購買歷史）

generateAiReply(caseId, kbIds[]) · 這是 Thermos 最完整的 RAG 範例

Prompt 組成三塊：

  1. 案件資訊：客戶名、會員等級、案件類型、通路、主旨、詳情、詢問商品、保固狀態
  2. 客戶購買歷史：SOQL 拉最近 5 筆已 Activated 訂單，含 OrderItem 商品名、類別，作為個人化依據
  3. 知識庫參考：從 Knowledge__kav（SOSL 搜出的 KB 文章 Title + Summary）

要求生 150-220 字繁中回覆、抬頭稱呼、依購買歷史自然帶入升級建議、結尾留聯絡方式。以下是 CBUSDO Case 500J200000NIuE1 的實際生成內容：

感謝您透過官網聯繫我們，並感謝您作為白金會員對 THERMOS 的支持！針對您詢問 JNL-352 保溫瓶墊圈的問題，JNL 系列的墊圈確實為通用設計，您可於官網直接搜尋「JNL 墊圈組」進行購買，單價約為 NT$80-180。建議您確認目前墊圈的狀況，若已使用超過 12 個月，建議更換以確保瓶身密封效果。 另外，我們也提供專用清潔刷，能有效清潔瓶內及配件，延長使用壽命。若有需要，您可一併選購，享受更完善的使用體驗！ 如需協助下單或有其他疑問，歡迎隨時透過官網或撥打客服專線與我們聯繫（0800-XXX-XXX）。再次感謝您的支持，祝您使用愉快！ 此致 THERMOS 客服中心 敬上

↑ 這段是真的 LLM 生的，不是寫死的樣板。可以在 Case.THERMOS_AI_Draft__c 查得到。「白金會員 / JNL-352 / 建議更換 12 個月」都是從 Case + Order data 拉出來個人化，不是預先寫在程式碼裡。

Reply Recommendations — 3 段不同語氣

getReplyRecommendations() · 一鍵生 3 個候選回覆

對同一份 KB 命中結果分別呼叫 Einstein 三次，每次 tone 不同：

Tone指令
Formal（專業直接）先致謝 → 直接說明處理方式 → 提供後續聯絡方式
Apology（致歉補償）主動致歉 → 說明處理 → 提供補償建議（100 點紅利）
Friendly（友善關懷）語氣親切 → 提供解決方案 → 表達持續關心

客服在 UI 上直接挑一段套用，或看完再叫 generateAiReply 生完整版。這是 Steven demo 常用的「AI 也能挑腔調」段。

Chat Agent · 2 個對話功能

Customer 360 Chat

ThermosCustomerChat.cls · Account 頁

對「這位客戶」問問題。SOQL 打包 Account + 訂單 aggregate + 通路分布 + Case 數 + Touch 記錄 → prompt。

用途：業務／老闆看客戶時，直接問 「這位客戶最有可能加購什麼？」「上次購買後怎麼再接觸？」

Q: 這位客戶的購買偏好？ A: 楊小姐為白金會員，累計消費 NT$ 82,400，主要購買 alfi 保溫瓶（占訂單金額 52%）及 THERMOS 食物罐。慣用 LINE OA 通路（占 6 成訂單），已綁定 LINE，近 30 天內尚無新訂單，建議在 LINE 推播 alfi 新色以再度啟動。

Business Chat（Home 頁）

ThermosBusinessChat.cls · Home 頁

面向 Liam ／ 董事長。跑全體 aggregate：會員數、VIP 數、LINE 綁定率、通路營收占比、Tier 分布、品牌品項、案件量、AI 加購命中數 → prompt。

用途：老闆問 「這個月哪個通路成長最快？」「AI 對加購貢獻多少？」

Q: 現在最值得投資的通路？ A: 官網通路貢獻 NT$ 1.4M（占 38%），LINE OA 佔比 24%，但 LINE 綁定率僅 42%。建議先把 LINE 綁定推到 60%，預估可再帶 NT$ 400K/季。

誠實揭露 · 這幾個 UI 看起來像 AI，其實是規則式

Knowledge Agent（客服 KM 搜尋頁）規則式

ThermosKnowledgeAgent.search()

Case 頁旁邊那個「AI Knowledge Agent」——它 UI 上寫「AI 搜尋」，但底層 沒呼叫 LLM：

  • 抽 [A-Z]{2,4}-\d{2,4} 型號 token（JNL-504、SK-3020）
  • 比對中英文關鍵字清單（保溫瓶／燜燒鍋／墊圈／THERMOS／alfi ...）
  • 加權評分：title +40、keywords +25、product code +25、body +10、熱門度加成 +5
  • 組合 aiAnswer 是 字串拼接的模板：「偵測到商品型號【JNL-504】，依知識庫最新版本，建議回覆：「...」」

為什麼這樣做？demo 現場延遲重要，keyword scoring 是 <100ms、可控且可預測；LLM 呼叫是 1-3 秒且會受 licence 影響。真正需要 LLM 精緻回覆的場景，用戶會走 Case 頁 generateAiReply（第 5 項），不是這個搜尋頁。寫給客戶時建議標「Einstein Search 風格 keyword scoring」而不是「生成式 AI」。

Insight Panel（客戶 360 洞察面板）規則式

THERMOSInsightService.getInsight()

Account 頁那個「AI 洞察 3 條」——也是規則式：

洞察怎麼算
消費輪廓SOQL aggregate 出 topBrand / topCategory / totalSpend
流失風險daysSinceLastOrder > 180 → 高；> 90 → 中；否則 → 低
通路偏好從 Account.Description tag 抽「慣用通路」＋ LINE 綁定狀態

要換成 LLM 也很快：ThermosCustomerChat.regenerateInsight(recordId, angle) 已經接好 EinsteinService，UI 側加個「重新用 AI 生成」按鈕就是了。目前規則式已能給穩定的敘述給老闆看，故 demo 預設走規則。

誠實揭露 · 這些 Salesforce 官方 AI feature「沒」實際用

功能狀態為什麼沒用
Agentforce for Service（Bot / GenAiPlanner metadata） 未使用 CBUSDO 有 8 個 OOTB Bot（Copilot for Salesforce、Sunny、Luna 等），但 Thermos APP 沒有自建 Bot。功能上我們自己在 Apex 拼「LLM + RAG + 8 個 Action」，等價於 Agentforce 但不走 Bot metadata。
Prompt Builder（GenAiPromptTemplate） 未使用 Org 上沒有 GenAiPromptTemplate metadata。所有 prompt 都以 String 動態組在 Apex（ThermosCase360Controller、ThermosCustomerChat、ThermosBusinessChat）。
好處：客戶能真的看到 prompt 內容並改；壞處：拿不到 Prompt Builder 的 Grounding UI。
Einstein Case Classification（OOTB） 未使用 需要歷史標記資料訓練，SDO 沒有足夠樣本。改用 LLM zero-shot（第 4 項）。
Einstein Reply Recommendations（OOTB） 未使用 OOTB 版需要 Digital Engagement channel 資料。我們自己實作三段語氣版（第 6 項）。
Einstein Service Analytics（Tableau） 未使用 demo 有 3 張 Report + 1 個 Dashboard（THERMOS_Dashboards），用原生 Report Builder，未接 Tableau。
Copilot Studio Actions（Invocable + Copilot config） 未使用 8 個 Agent Action（executeAgentAction）以 @AuraEnabled 供 LWC 呼叫，沒登記成 Copilot Action。
Data Cloud / Einstein Predictive AI 未使用 Thermos 是純 Sales+Service Cloud demo，不涉及 Data 360／預測模型。若要延伸，會另建 uniicreative 或配方時代那類 demo。
結論一句話：
Thermos APP 的 AI ＝ Einstein LLM（Trust Layer）＋ RAG（Knowledge__kav + Order + Account）＋ 8 個真操作 Case/Account 的 Action。功能上等於 Agentforce for Service，但實作走 Apex ConnectApi 而不是 Bot / GenAiPlanner metadata。Prompt Builder、Einstein OOTB feature 都沒用到。
